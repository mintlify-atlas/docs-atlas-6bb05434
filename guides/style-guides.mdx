---
title: "Style guides"
description: "Community and company Swift style guides to help you write clean, consistent code."
---

A consistent coding style makes your Swift code easier to read, review, and maintain. These guides cover naming, formatting, and idiomatic patterns — each reflecting the values and scale of the organization that wrote it.

<CardGroup cols={2}>
  <Card
    title="Airbnb Swift Style Guide"
    icon="house"
    href="https://github.com/airbnb/swift"
  >
    **Publisher:** Airbnb

    Airbnb's official guide for Swift code across their iOS codebase. It emphasizes clarity and predictability, with rules on naming, access control, optionals, and SwiftUI patterns. Well-suited for teams working on large consumer apps.
  </Card>
  <Card
    title="Google Swift Style Guide"
    icon="google"
    href="https://google.github.io/swift/"
  >
    **Publisher:** Google

    Built on top of Apple's Swift standard library style and refined through production use across multiple Google Swift projects. It provides detailed formatting rules, documentation conventions, and guidance on when to prefer certain language features over others.
  </Card>
  <Card
    title="LinkedIn Swift Style Guide"
    icon="linkedin"
    href="https://github.com/linkedin/swift-style-guide"
  >
    **Publisher:** LinkedIn

    LinkedIn's official style guide, shaped by the demands of a large engineering organization shipping a high-traffic iOS app. It covers code organization, spacing, and patterns that keep a shared codebase navigable at scale.
  </Card>
  <Card
    title="Kodeco (Ray Wenderlich) Swift Style Guide"
    icon="code"
    href="https://github.com/kodecocodes/swift-style-guide"
  >
    **Publisher:** Kodeco (formerly Ray Wenderlich)

    A widely referenced guide written for the Swift tutorial community. It prioritizes readability and is grounded in practical examples. A strong choice if you are learning Swift or building a style baseline for a smaller team.
  </Card>
</CardGroup>

<Note>
  No single style guide is universally correct. Pick one that fits your team's context, apply it consistently, and enforce it with a linter like [SwiftLint](https://github.com/realm/SwiftLint) to remove subjective debates from code review.
</Note>
